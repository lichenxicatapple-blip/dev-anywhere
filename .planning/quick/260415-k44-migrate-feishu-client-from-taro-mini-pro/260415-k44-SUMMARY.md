# Quick Task 260415-k44: Feishu to React SPA Migration Plan Summary

v3 rewrite of migration plan: removed all artificial resource limits, eliminated premature throttling, trust modern developer hardware.

## What Was Done

### Task 1: Rewrite MIGRATION-PLAN.md v3

Rewrote the migration plan from scratch to eliminate v2's problems:

**Removed artificial limits:**
- ReplayBuffer changed from 1MB ring buffer to growable array of Uint8Array chunks with no fixed upper bound
- Removed 256KB bufferedAmount threshold for WebSocket forwarding
- Removed "drop intermediate frames" policy -- never drop data
- Removed 100KB/s output sampling threshold on proxy side
- Removed xterm.js bundle size as a "risk" item (300-400KB for a terminal app is not a risk)
- Removed maxEntries from Workbox cache config

**Removed unnecessary abstractions:**
- Removed Navigation Helper section -- react-router useNavigate() is sufficient

**Simplified exception handling (Section 5.4):**
- Reconnection: replay ALL buffered data, let xterm.js scrollback setting be the natural display limit
- High-speed output: trust xterm.js built-in write batching (requestAnimationFrame), no proxy-side throttling
- Backpressure: rely on TCP flow control, disconnect dead clients, don't silently drop frames
- UTF-8/ANSI boundaries: xterm.js handles natively, no special handling needed

**Kept intact from v2:**
- Architecture overview and data flow diagrams
- Project structure
- Design tokens and CSS migration strategy
- Component migration map (all 17 components)
- Taro API to Web API mapping table
- Protocol changes (pty_data message type)
- Phased execution plan (A-H)
- Deployment scenarios
- Dependencies table
