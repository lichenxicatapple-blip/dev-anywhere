# Quick Task 260415-k44: Feishu to React SPA Migration Plan Summary

v4 rewrite: persistent snapshots via restored EventStore, binary WebSocket frames from day one, no artificial limits, no unnecessary abstractions.

## What Was Done

### Task 1: Rewrite MIGRATION-PLAN.md v4

Rewrote the migration plan incorporating all v3->v4 changes:

**Added disk persistence (critical):**
- Restored EventStore design from commit b05bec2: binary format (CCAE header + length-prefixed records), 1s buffer window, gzip archival
- Restored TerminalTracker: @xterm/headless + serialize addon for periodic snapshots (working->idle + every 100 events)
- Disk layout: `~/.cc-anywhere/data/{sessionId}/{events.bin, events.bin.gz, snapshot.bin}`
- Proxy restart recovery: load latest snapshot from disk, replay subsequent events, resume normal operation
- Session history playback: read events.bin, feed to xterm.js at any speed

**Binary WebSocket frames from day one:**
- PTY data transmitted as binary WebSocket frames with sessionId prefix (2-byte length + UTF-8 sessionId + raw bytes)
- Relay distinguishes binary vs text frames at WebSocket protocol level
- Removed the "start with base64+JSON, optimize later" approach from v3
- Removed the "Binary Data Encoding" section that hedged toward JSON+base64

**Removed artificial limits (carried from v3):**
- No hardcoded buffer sizes, no output rate thresholds, no frame dropping policies
- No ReplayBuffer ring buffer with max size (replaced by EventStore disk persistence)

**Removed toWebPath():**
- Route paths changed directly: `/pages/chat/index` -> `/chat`
- Nav object uses `useNavigate()` with simple paths, no conversion function

**Removed fake performance concerns:**
- No xterm.js bundle size risk
- No mobile rendering performance concern
- No "optimize to binary frames later" hedging

**Reconnection redesigned around snapshots:**
- Proxy sends latest snapshot + all events since snapshot from disk
- Client loads snapshot via serialize addon, then writes subsequent events
- No "tail guessing" or byte offset estimation -- snapshots solve the starting point cleanly

**Kept intact from v3:**
- Architecture overview and data flow diagrams (updated for new pipeline)
- Project structure
- Design tokens and CSS migration strategy
- Component migration map (all 17 components)
- Taro API to Web API mapping table
- Phased execution plan (A-H, Phase D expanded for persistence + binary pipeline)
- Deployment scenarios
- Dependencies table (added @xterm/addon-serialize)
