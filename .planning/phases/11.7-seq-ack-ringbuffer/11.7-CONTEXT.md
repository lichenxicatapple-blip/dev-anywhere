# Phase 11.7: seq/ack + proxy 单层 ring buffer - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning
**Source:** LinkShell 协议层调研（单层简化变体）+ cc_anywhere 断线重连闪屏痛点

<domain>
## Phase Boundary

为 proxy → relay → client 的 PTY 数据流加**seq/ack + 单层 ring buffer**，消除断线重连时的 `term.reset() + 全量 snapshot` 闪屏。**关键选择**：buffer 放在 proxy 侧单层，**不抄 LinkShell 的三层**（CLI + gateway + client），避免跨层一致性协调成本。

**范围覆盖**：
- proxy 侧：per-session ring buffer（1000 条 envelope 或按字节上限），proxy → relay 的每条 envelope 打 seq（envelope 顶层字段）
- relay 透传保留"无状态"语义（memory `feedback_relay_无状态`），不缓存
- shared schema：envelope 加 `seq` 字段；新增 `session.resume` / `session.ack` 控制消息
- client 侧：
  - 收到每条 envelope 更新 `lastReceivedSeq`；每 ~500ms 批量 `ack(seq)` 回 proxy
  - reconnect 时发 `session.resume {lastAckedSeq}` 代替全量 `session_subscribe`
  - resume 返回的增量流直接 `term.write` 无需 reset
- proxy 收到 resume 请求 → 从 ring buffer 回 `seq > lastAckedSeq` 的 envelope
- fallback：如果 lastAckedSeq 已被 ring buffer 滚出 → proxy 回全量 snapshot（现有流程），client 照旧 reset + write

**范围外**：
- 三层 scrollback（CLI 端 + gateway 端 + client 端）——砍，单层够
- 冷启动恢复（PWA 被杀后从 IndexedDB 恢复 xterm buffer）——Phase 11.7.x 或更后（本 phase 只解 Case A：页面没被杀）
- Hook 事件进 ring buffer ——本 phase **不包括**（Hook envelope 可以进 ring buffer 但不额外设计）。保险起见 Hook 走独立路径，等这个 phase 落地后再评估
- scroll UX（Phase 11.6）

**允许跨包改动**：apps/proxy + apps/relay（只改透传不改状态） + apps/web + packages/shared。

</domain>

<decisions>
## Implementation Decisions

### 协议层

- **D-01**: **seq 字段加到 envelope 顶层**（不是 payload 里）。给每个 session 一个独立单调递增计数器，proxy 侧 `SessionSeqCounter` 管理。
- **D-02**: **seq 类型 `number`**（不用 BigInt），上限 `2^53 - 1`。Claude session 单次会话最多产几 M envelope，远不到上限。
- **D-03**: **binary PTY 帧是否带 seq**：**带**。binary 帧现在没进 envelope 结构（是 `sendBinary` 的 raw buffer），本 phase 改造为**包装进 envelope**（envelope type = `terminal.pty_output`，payload 是 base64 或二进制，视能力而定），统一走 seq/ack。
- **D-04**: **ack 批量策略**：client 侧每 500ms 或收到 N=50 条 envelope 时发一次 `session.ack {sessionId, upTo: seq}`。不每条都 ack，减少上行流量。
- **D-05**: **ack 单调性**：`upTo` 必须大于上次 ack 的值，proxy 侧校验；重复/乱序 ack 忽略。
- **D-06**: **resume 协议**：
  - client 断线重连后 → `session.resume {sessionId, lastAckedSeq}` 代替现有的 `session_subscribe`
  - proxy 收到 → 查 ring buffer：
    - 命中：回 `session.resume_response {ok: true}` + 按序重发 `seq > lastAckedSeq` 的 envelope
    - 未命中（lastAckedSeq 太老，已滚出 buffer）：回 `session.resume_response {ok: false, reason: "buffer_overflow"}` + 触发现有 `session_subscribe` 全量 snapshot 流程（client 端 reset + write 全量）
    - 未知 sessionId：`ok: false, reason: "session_not_found"`，client 侧按"新会话"处理

### Ring Buffer 设计

- **D-07**: **per-session buffer**：每个 sessionId 独立 ring。terminate session 时整个 buffer 释放。
- **D-08**: **容量**：默认 **1000 条 envelope**，**不按字节上限**（实现简单）。后续若发现高频 output 撑爆内存再改成 `max(1000 条, 4MB)` 之类的双约束。
- **D-09**: **淘汰策略**：FIFO（超容量丢最老）。trim 触发点仅在 enqueue 时，不做基于 ack 的主动 trim（避免复杂度）。
- **D-10**: **实现位置**：`apps/proxy/src/serve/session-ring-buffer.ts`（新文件），被 RelayConnection 或 SessionManager 使用。具体归属留给 planner。
- **D-11**: **持久化**：**不持久化**。proxy 重启 = buffer 清空 = client 重连时只能走全量 snapshot。这对"自己用爽"够用。

### Client 侧

- **D-12**: **`lastAckedSeq` 存储**：**内存 + localStorage 双存**。内存用于快速 ack；localStorage 用于冷启动恢复（Case B 极端场景部分救回来，不完整但聊胜于无）。
- **D-13**: **resume 失败降级**：`ok: false` 时退回现有全量 snapshot 流程。不新增 UI 提示，对用户透明（只是看到一次闪屏）。
- **D-14**: **Wake recovery 联动**（Phase 11.5 之前的 commit）：wake 触发的 reconnect 完成后，自动发 resume；如果这条连接的 sessionId 尚未选中（用户在 ProxySelect 页），不发 resume（没业务会话 context）。

### Relay

- **D-15**: **relay 不读 seq**。透传 proxy → client 方向的 envelope，不做任何拦截/缓存/校验。relay 仅做"找到目标 client"并转发。
- **D-16**: **relay 也不读 ack / resume**。client 发的 `session.ack` / `session.resume` 走客户端 → proxy 的现有转发链路（从 `CLIENT_TO_PROXY_TYPES` allowlist 放行）。memory `project_relay_control_allowlist` 提醒的要点：加新 type 要记得加 allowlist。

### 测试

- **D-17**: **单测**：ring buffer 的 enqueue / trim / replay-after-seq / overflow 返回 false。
- **D-18**: **集成测**：断线重连场景 1：ack 落后不多（ring buffer 能覆盖）→ 无 reset 续上。场景 2：ack 落后太多 → 降级全量。场景 3：session 被 terminate → resume 回 `session_not_found`。
- **D-19**: **手动验收**：真机锁屏 10 秒/1 分钟/10 分钟再解锁，分别验证 Case A 无闪屏、Case B 降级时只闪一次。

</decisions>

<specifics>
## Specific Ideas

### 新 envelope types

```ts
// shared/src/schemas/relay-control.ts 追加

// proxy → client: 原有的 PTY 字节流，改成包进 envelope
"terminal.pty_output"     // payload: { data: Uint8Array / base64 }

// client → proxy
"session.ack"              // payload: { sessionId: string, upTo: number }
"session.resume"           // payload: { sessionId: string, lastAckedSeq: number }

// proxy → client
"session.resume_response"  // payload: { sessionId: string, ok: boolean, reason?: string, seq?: number (当前最新 seq) }
```

### allowlist 更新

- `PROXY_TO_CLIENT_TYPES` 加：`terminal.pty_output`（如果之前是 binary 直发），`session.resume_response`
- `CLIENT_TO_PROXY_TYPES` 加：`session.ack`，`session.resume`

### Binary 包进 envelope 的取舍

目前 `sendBinary` 是 raw buffer + 自定义 framing（`sidLen + sessionId + ptyData`，见 `websocket.ts:126-136` dispatchBinary）。本 phase 有两个路径：
- **A. 继续 binary**：给 binary 帧加一个前缀 header 带 seq（紧凑，性能好）
- **B. 改 envelope**：PTY 帧进 envelope payload（base64 或 Uint8Array），统一走 JSON 控制流

planner 评估：**A 更紧凑**（binary 帧尺寸小延迟敏感）；**B 更统一**（一套协议通吃）。倾向 A——binary 加前缀字节 seq header。保留现在的 binary subscribeBinary 路径，dispatchBinary 里 parse 出 seq。

</specifics>
