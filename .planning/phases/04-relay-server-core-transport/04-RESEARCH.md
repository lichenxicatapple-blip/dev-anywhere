# Phase 4: Relay Server - Core Transport - Research

**Researched:** 2026-04-06
**Domain:** WebSocket relay server, message routing, Docker deployment, Taro spike
**Confidence:** HIGH

## Summary

Phase 4 implements the WebSocket relay server that bridges the local proxy service and remote clients (Feishu mini program). The relay accepts outbound WebSocket connections from both proxy and remote clients, routes messages between them based on proxyId registration, and validates messages against the shared MessageEnvelope schema. The phase also includes a Taro spike to validate Feishu mini program WebSocket connectivity before building the full relay.

The existing codebase provides strong foundations: `@cc-anywhere/shared` exports `MessageEnvelopeSchema` with 16 message types, `buildMessage` for envelope construction, and `ErrorCode` for error handling. The relay server is a new `apps/relay` package with only a placeholder `index.ts`. The proxy service (`apps/proxy/src/serve.ts`) needs modification to add an outbound WebSocket connection to the relay.

**Primary recommendation:** Use `ws` with `noServer: true` attached to an Express HTTP server. Express handles health check endpoints; `ws` handles WebSocket upgrade with URL-path routing (`/proxy` and `/client` endpoints). Relay is a stateless message router -- it maintains an in-memory registry of proxy connections and routes messages without persisting anything. The shared package needs new relay-specific control message types (proxy registration, proxy list, proxy selection).

### Pattern 1: ws noServer + Express HTTP Server

**What:** Create ws WebSocketServer with `noServer: true`, attach to Express's HTTP server via the `upgrade` event. Route by URL pathname: `/proxy` for proxy connections, `/client` for remote client connections. [CITED: github.com/websockets/ws]

**When to use:** When you need HTTP endpoints (health checks) alongside WebSocket on the same port.

**Example:**
```typescript
// Source: ws official README (github.com/websockets/ws)
import express from "express";
import { createServer } from "node:http";
import { WebSocketServer } from "ws";

const app = express();
const server = createServer(app);

const proxyWss = new WebSocketServer({ noServer: true });
const clientWss = new WebSocketServer({ noServer: true });

server.on("upgrade", (request, socket, head) => {
  const { pathname } = new URL(request.url!, `http://${request.headers.host}`);

  if (pathname === "/proxy") {
    proxyWss.handleUpgrade(request, socket, head, (ws) => {
      proxyWss.emit("connection", ws, request);
    });
  } else if (pathname === "/client") {
    clientWss.handleUpgrade(request, socket, head, (ws) => {
      clientWss.emit("connection", ws, request);
    });
  } else {
    socket.destroy();
  }
});
```

### Pattern 2: Proxy Registration + Client Routing

**What:** Relay maintains two in-memory maps:
1. `proxyRegistry: Map<proxyId, WebSocket>` -- proxy connections indexed by proxyId
2. `clientBindings: Map<WebSocket, proxyId>` -- which proxy each remote client is targeting

Proxy sends a registration message on connect with its proxyId. Remote client requests proxy list, then selects one. All subsequent messages route through the selected binding.

**When to use:** Always -- this is the core relay routing mechanism per D-05/D-06/D-07.

**Example:**
```typescript
// Relay 内部注册表
interface RelayRegistry {
  // proxyId -> WebSocket 连接
  proxies: Map<string, WebSocket>;
  // remote client WebSocket -> 选定的 proxyId
  clientBindings: Map<WebSocket, string>;
}

// Proxy 连接时发送注册消息
// Remote client 连接后获取 proxy 列表，选择一个 proxyId
// 后续消息根据 binding 双向转发
```

### Pattern 3: Heartbeat with isAlive Flag

**What:** Server sends ping frames at fixed intervals. If no pong received before next interval, connection is dead and gets terminated. [CITED: ws README + oneuptime.com heartbeat guide]

**Recommended parameters:**
- Heartbeat interval: 30 seconds
- Dead connection detection: if no pong within 30s, terminate on next check
- These parameters apply to both proxy and client connections

**Example:**
```typescript
// Source: ws official README heartbeat example
const HEARTBEAT_INTERVAL = 30_000;

function setupHeartbeat(wss: WebSocketServer): NodeJS.Timeout {
  return setInterval(() => {
    for (const ws of wss.clients) {
      if ((ws as any).isAlive === false) {
        ws.terminate();
        continue;
      }
      (ws as any).isAlive = false;
      ws.ping();
    }
  }, HEARTBEAT_INTERVAL);
}

// 连接建立时
wss.on("connection", (ws) => {
  (ws as any).isAlive = true;
  ws.on("pong", () => { (ws as any).isAlive = true; });
});
```

### Pattern 4: Message Validation + Transparent Forwarding

**What:** Relay parses incoming WebSocket text frames as JSON, validates with `MessageEnvelopeSchema.safeParse()`. Valid messages are forwarded to the target connection as-is (re-serialized JSON string). Invalid messages are dropped with a pino log entry.

**When to use:** All message forwarding per D-08.

### Anti-Patterns to Avoid

- **Relay as state owner:** Relay should NOT persist session state, message history, or maintain its own seq counters. It is a dumb router. Proxy owns session state (per architecture decision).
- **Parsing message content for routing:** Relay routes based on connection identity (which WebSocket sent it), not by inspecting message payload fields. The proxyId binding determines where messages go.
- **Synchronous message processing:** Use event-driven async patterns. Never block the event loop parsing or validating messages.
- **Single WebSocketServer for both proxy and client:** Use separate WebSocketServer instances with noServer + URL path routing. This cleanly separates the two connection types and their lifecycle handlers.

## Shared Package Extensions Required

Phase 4 needs relay-specific control messages that do NOT exist in the current shared package. These are out-of-band control messages for relay connection management, distinct from the 16 existing MessageEnvelope types.

| New Message | Direction | Purpose |
|-------------|-----------|---------|
| `proxy_register` | Proxy -> Relay | Proxy announces its proxyId on connect |
| `proxy_list_request` | Client -> Relay | Client asks for online proxy list |
| `proxy_list_response` | Relay -> Client | Relay returns online proxy list |
| `proxy_select` | Client -> Relay | Client selects a proxyId to interact with |
| `relay_error` | Relay -> Any | Relay-originated error (invalid message, unknown proxy, etc.) |

**Design decision:** These control messages should be a separate schema (e.g., `RelayControlSchema`) in the shared package, NOT added to `MessageEnvelopeSchema`. Reason: MessageEnvelope is the application-level protocol between proxy and client (16 message types with seq, sessionId, timestamp, source, version). Relay control messages are transport-level -- they don't carry sessionId or seq because they exist outside the session context.

```typescript
// packages/shared/src/schemas/relay-control.ts
const RelayControlSchema = z.discriminatedUnion("type", [
  z.object({ type: z.literal("proxy_register"), proxyId: z.string() }),
  z.object({ type: z.literal("proxy_list_request") }),
  z.object({
    type: z.literal("proxy_list_response"),
    proxies: z.array(z.object({ proxyId: z.string() })),
  }),
  z.object({ type: z.literal("proxy_select"), proxyId: z.string() }),
  z.object({
    type: z.literal("relay_error"),
    code: z.string(),
    message: z.string(),
  }),
]);
```

Relay receives a WebSocket text frame, first tries `RelayControlSchema.safeParse()`. If it matches, handle as control message. Otherwise, try `MessageEnvelopeSchema.safeParse()` and forward to the bound target.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WebSocket server | Raw `http.createServer` upgrade handling | `ws` WebSocketServer | RFC 6455 compliance, ping/pong, backpressure, proven at scale |
| JSON schema validation | Manual `typeof` checks | `zod` safeParse | Already have schemas in shared package. Type inference. Consistent error messages. |
| Unique ID generation | `Math.random().toString(36)` | `nanoid` | Collision-resistant, URL-safe, configurable length |
| Structured logging | `console.log` with JSON.stringify | `pino` | 30x faster than console. Structured. Log levels. File transport. Already used in proxy. |
| HTTP routing | Raw `http.createServer` request handling | Express | Middleware, error handling, route patterns for health/status endpoints |
| SSL/TLS termination | Node.js `https.createServer` with cert management | Nginx reverse proxy | Nginx handles cert renewal, connection buffering, and is the standard for production WSS |

## Common Pitfalls

### Pitfall 1: Taro 4 + Lark Plugin Compatibility

**What goes wrong:** `@tarojs/plugin-platform-lark` v1.1.5 was built against Taro 3.x. Using it with Taro 4.1.11 may cause build failures or runtime errors in the Feishu mini program. [VERIFIED: npm registry -- plugin devDeps show @tarojs/service ^3.3.0]

**Why it happens:** The plugin's last publish was over a year ago. Taro 4 introduced breaking changes in the plugin API.

**How to avoid:** This is exactly why D-01 exists -- the Taro spike validates this before investing more effort. If the plugin fails with Taro 4, fallback options:
1. Pin Taro to 3.6.x for the mini program package only
2. Use native Feishu TTML without Taro (worse DX but guaranteed to work)
3. Fork and patch the plugin

**Warning signs:** Build errors mentioning `getPluginPlatform`, `modifyBuildTempFileContent`, or other Taro plugin lifecycle hooks.

### Pitfall 2: WebSocket Message Ordering vs TCP Ordering

**What goes wrong:** Developers assume WebSocket messages arrive in send order. They do (WebSocket is over TCP), but only within a single connection. If relay re-serializes and forwards, the forwarded message order depends on the relay's processing order, which is guaranteed single-threaded in Node.js. No pitfall here as long as relay uses synchronous forwarding within the event loop tick.

**Why it happens:** Confusion between TCP ordering (guaranteed) and application-level ordering across reconnections (not guaranteed -- Phase 5's problem).

**How to avoid:** Phase 4 seq numbers are populated but not checked. Just ensure `ws.send()` calls for forwarding happen synchronously within the message event handler -- don't use `await` between receiving and forwarding.

**Warning signs:** Messages arriving out of order during normal operation (not reconnection). This would indicate a bug in relay forwarding logic.

### Pitfall 3: Express 5 + ws handleUpgrade Integration

**What goes wrong:** Express 5.x changed the internal HTTP server creation. The `app.listen()` return value is still a standard `http.Server`, but some middleware may interfere with the upgrade event.

**Why it happens:** Express middleware that doesn't call `next()` can swallow upgrade requests.

**How to avoid:** Create `http.createServer(app)` explicitly and listen on that, rather than using `app.listen()`. This ensures you have direct access to the HTTP server's `upgrade` event. Use `server.on('upgrade', ...)` before any middleware could interfere.

### Pitfall 4: Docker Build Context and Monorepo

**What goes wrong:** The Dockerfile for `apps/relay` needs access to `packages/shared` (workspace dependency). If the Docker build context is `apps/relay/`, the shared package isn't available.

**Why it happens:** Docker build context is typically the directory containing the Dockerfile. In a monorepo, dependencies live outside this directory.

**How to avoid:** Set the Docker build context to the monorepo root. Use `.dockerignore` to exclude unnecessary files. The Dockerfile references `packages/shared/` and `apps/relay/` relative to root.

```dockerfile
# Build context: monorepo root
# docker build -f apps/relay/Dockerfile .
COPY packages/shared/ packages/shared/
COPY apps/relay/ apps/relay/
COPY pnpm-workspace.yaml pnpm-lock.yaml package.json ./
```

### Pitfall 5: Nginx WebSocket Proxy Timeout

**What goes wrong:** Nginx's default `proxy_read_timeout` is 60 seconds. WebSocket connections that are idle for 60 seconds get closed by Nginx.

**Why it happens:** Nginx treats WebSocket connections like HTTP connections by default.

**How to avoid:** Set `proxy_read_timeout` and `proxy_send_timeout` to a high value (86400s = 24 hours). The ws heartbeat (30s ping/pong) keeps the connection active at the application level, but Nginx needs to know not to close idle connections. [CITED: websocket.org/guides/infrastructure/nginx]

### Pitfall 6: Proxy Service Shutdown Doesn't Close Relay Connection

**What goes wrong:** When proxy service shuts down (SIGTERM), it cleans up local sockets and worker processes but doesn't cleanly close its WebSocket connection to the relay. The relay sees a dead connection only after the next heartbeat timeout (up to 30s delay).

**Why it happens:** Phase 3's shutdown handler in `serve.ts` doesn't know about the relay connection.

**How to avoid:** When adding relay connection to proxy's `serve.ts`, also add `ws.close()` to the existing `shutdown()` function. The relay should remove the proxy from its registry immediately on WebSocket `close` event.

### Message Validation and Forwarding

```typescript
// apps/relay/src/router.ts
import { MessageEnvelopeSchema } from "@cc-anywhere/shared";
import type { WebSocket } from "ws";
import type pino from "pino";

// 来自 relay-control schema 的控制消息处理
// MessageEnvelope 消息的透传转发
export function handleProxyMessage(
  data: string,
  proxyId: string,
  registry: RelayRegistry,
  logger: pino.Logger,
): void {
  const parsed = MessageEnvelopeSchema.safeParse(JSON.parse(data));
  if (!parsed.success) {
    logger.warn({ proxyId, error: parsed.error.message }, "Invalid message from proxy, discarding");
    return;
  }

  // 转发给绑定到该 proxy 的所有 remote client
  for (const [clientWs, boundProxyId] of registry.clientBindings) {
    if (boundProxyId === proxyId && clientWs.readyState === 1) {
      clientWs.send(data);
    }
  }
}
```

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | Deferred (D-04) | No auth in Phase 4. Explicitly deferred. |
| V3 Session Management | No | Relay is stateless, session state owned by proxy |
| V4 Access Control | Deferred (D-04) | No access control in Phase 4. Open relay by design for now. |
| V5 Input Validation | Yes | zod MessageEnvelopeSchema.safeParse() for all forwarded messages |
| V6 Cryptography | Yes (TLS) | Nginx TLS termination for WSS. No application-level crypto needed. |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unauthenticated WebSocket access | Spoofing | Deferred to post-Phase 5. Phase 4 relay is intentionally open (D-04). |
| Malformed message injection | Tampering | zod schema validation. Invalid messages dropped, not forwarded (D-08). |
| Message content in server logs | Information Disclosure | pino logs metadata only (proxyId, message type), never payload content. |
| WebSocket connection flooding | DoS | Heartbeat + dead connection cleanup. Rate limiting deferred. |
| Man-in-the-middle on ws:// | Tampering / Info Disclosure | Nginx TLS termination enforces WSS (D-13). |
