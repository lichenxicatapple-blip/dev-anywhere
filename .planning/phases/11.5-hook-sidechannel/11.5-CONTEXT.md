## Phase Boundary

<decisions>
## Implementation Decisions

### 架构选择

- **D-01**: **单一 HTTP endpoint 在 serve daemon**（一台机器一份），不用 LinkShell 的"每 bridge-session 起一份 HTTP server + marker 过滤"。理由：我们 serve 本来就是 daemon，多 terminal 共享一份 endpoint 更简洁；sessionId 路由足够隔离。
- **D-02**: **`CC_ANYWHERE_SESSION_ID` env 透传 + URL query** 做路由：terminal 在 pty.spawn 时注入 env；hook curl 命令里 `--data-urlencode "sessionId=$CC_ANYWHERE_SESSION_ID"` 带上；serve 的 HTTP handler 根据 sessionId 路由到对应 terminal socket。
- **D-03**: **HTTP endpoint 端口选择**：`127.0.0.1:<random-port>` 每次 serve 启动新建（避免端口占用冲突），端口号写进 `~/.cc-anywhere/hook-port` 文件，terminal spawn 时读取注入到 hook curl 命令里。
- **D-04**: **Hook 事件协议**：sever HTTP endpoint 收到 payload 后 → 包装成 `terminal.hook_event` envelope → relayConnection.sendEnvelope → relay 透传到 client。
- **D-05**: **PermissionRequest 挂住响应**：HTTP handler 给每个 PermissionRequest 生成 `request_id`，注册到 pending map，转成 envelope 发 client，等 client 回 `terminal.permission_decision {request_id, decision}` 后 resolve 原 HTTP 响应体 `{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"|"deny"}}}`。timeout 默认 **24 小时**（对齐 LinkShell）。
- **D-06**: **settings.json 注入策略**：append 模式（不覆盖用户现有 hooks），每条注入行带 marker 注释 `# cc-anywhere-hook-marker:{daemonId}`；serve 启动时先清理所有自己旧 marker 的行再写入；daemonId 从 `~/.cc-anywhere/daemon-id` 读取（serve 首次启动生成 nanoid，持久化）。
- **D-07**: **退出清理**：serve 收到 SIGTERM/exit 时按 marker 行清理 settings.json。异常崩溃依赖下次 serve 启动时的 stale marker 清理逻辑兜底。

### 事件设计

- **D-08**: **Hook 事件 subtype**：沿用 Claude Code hook event 名字 `PreToolUse / PostToolUse / PermissionRequest / Stop / UserPromptSubmit / SessionStart`，作为 envelope payload 里的 `hookEventName` 字段。
- **D-09**: **payload 字段**：`{ hookEventName, sessionId, toolName?, toolInput?, transcriptPath?, timestamp }`。`toolName / toolInput` 仅在 PreToolUse / PermissionRequest / PostToolUse 有；其他事件这两个字段为 undefined。
- **D-10**: **`terminal.status` 广播**：serve 根据 hook 事件流推导当前 phase（`idle / thinking / tool_use / waiting / error`），每次变化 emit 一条 `terminal.status` envelope。这给 client 做"当前在干嘛"的 rich UI 基础。

### UI / 交互

- **D-11**: **审批 UI 在本 phase 做临时简版**：toast 式通知 + Approve / Deny 按钮，PTY 终端上方浮层展示 tool_name 和 tool_input（截断到 200 字符）。样式精细化交给 Phase 11.6 scroll 重构时一起做。
- **D-12**: **JSON 模式已有 `tool-approval-manager` 不复用**：Hook 审批是独立于 stdio 的路径，单独 manager（`hook-approval-manager`）。避免改动 JSON 模式现有链路。
- **D-13**: **OSC-based 状态推导保留**：`osc-extractor.ts` 继续跑，作为 hook 的补充。hook 事件优先（精确），OSC 事件兜底（如果用户没装 hook 或 Claude CLI 版本过老）。

### 鲁棒性

- **D-14**: **Hook endpoint 僵死不阻塞 claude**：HTTP handler 必须快返（非 PermissionRequest 场景 <50ms），否则会拖慢 claude 的下一步。PermissionRequest 挂住是**协议要求**（claude 同步等），不算阻塞。
- **D-15**: **settings.json 写失败降级**：如果 `~/.claude/settings.json` 权限不够或磁盘满，serve 启动时 log warn 并**继续跑**（hook 不可用但 proxy 其他功能正常）。client 侧在 `bridge_status` 增加 `hookAvailable: boolean` 标志。
- **D-16**: **PermissionRequest 孤儿**：如果 client 断线且 pending 审批超时（默认 24h），hook HTTP 响应回 `{"decision":{"behavior":"deny"}}` + log warn，避免 claude 永远挂着。

</decisions>

<specifics>
## Specific Ideas

### hook 命令模板（注入 settings.json）

```json
{
  "PermissionRequest": [{
    "hooks": [{
      "type": "command",
      "command": "curl -s -X POST --max-time 86400 \"http://127.0.0.1:$CC_ANYWHERE_HOOK_PORT/hook?sessionId=$CC_ANYWHERE_SESSION_ID&event=PermissionRequest\" --data-binary @- ; # cc-anywhere-hook-marker:<daemonId>",
      "timeout": 86400
    }]
  }],
  "PreToolUse": [{ ... timeout: 5 ... }],
  "PostToolUse": [{ ... }],
  ...
}
```

环境变量 `CC_ANYWHERE_HOOK_PORT` 和 `CC_ANYWHERE_SESSION_ID` 由 `pty.spawn` 的 `env` 选项注入 claude 进程。

### daemonId 生成

- `~/.cc-anywhere/daemon-id` 文件（首次 serve 启动生成 nanoid(12)，复用 proxyId 的 `loadOrCreate` 模式）
- 用途：settings.json marker 注释，区分"本机 proxy 写的" vs "别的 LinkShell-like 工具写的"
