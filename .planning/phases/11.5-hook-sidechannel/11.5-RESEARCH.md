# Phase 11.5: Hook 侧信道 - Research

**Researched:** 2026-04-21
**Domain:** Claude Code hooks (settings.json 注入) + proxy/serve 本地 HTTP endpoint + envelope schema 扩展 + web 审批 UI
**Confidence:** HIGH（hooks 官方 schema 已 cited；LinkShell 参考实现已 read；本仓所有落点文件已定位）

## Summary

本 phase 把 PTY 模式从"只看得见 ANSI 流"升级为"看得见结构化事件 + 可远程审批"。技术路径已由 CONTEXT.md 锁死：
1. serve daemon 起一份 `127.0.0.1:<random-port>` 的 HTTP endpoint（不是每 terminal 一份）；
2. terminal.ts 在 `pty.spawn` 时把 `CC_ANYWHERE_SESSION_ID` + `CC_ANYWHERE_HOOK_PORT` 注入 claude 进程 env；
3. serve 首次启动时往 `~/.claude/settings.json` 写 hooks（append，带 marker 注释），退出时按 marker 清掉；
4. hook 收到事件 → 包成 envelope → relay 透传 → client；`PermissionRequest` 同步挂住 HTTP 响应等 client 决策。

关键风险不在"做不做得出来"——LinkShell 已验证这套路径。关键风险是：
- **settings.json 是外部可变状态**（用户 / 其他工具 / claude 自己都可能改），本 phase 的 marker-based remove 逻辑必须做成对"当下文件真实内容"的 filter，不能假设上次注入的结构还在；
- **hook 进程的 env 继承**：`pty.spawn(env: {...})` → claude 进程 → hook 子进程（由 claude 以 `/bin/sh -c` 执行 curl），继承关系在 POSIX 下成立；Claude Code 官方文档明确 "Hooks inherit Claude Code's environment during execution" [CITED: code.claude.com/docs/en/hooks]；
- **并发写 settings.json 没有文档保证**[CITED: code.claude.com/docs/en/hooks]，必须用 atomic rename（`writeFileSync` tmp → `renameSync`）而不是直接 in-place 覆写。

**Primary recommendation:** 按 CONTEXT.md 的 D-01 ~ D-16 全部落地，不发明新方案；参考 LinkShell `bridge-session.ts:867-1419` 的 hook server / setupClaudeHooks / removeHookEntries / drainPendingPermissions 四块代码直接翻写，但结构上对齐本仓的 `ToolApprovalManager`（纯数据管理器）+ `RelayConnection.sendRaw/sendEnvelope` 分层。

## HTTP Framework Choice

**结论**：用 `node:http`，不上 express/fastify。

理由：
- 绑定 `127.0.0.1`，不对外，无需中间件栈
- 只有一条路由 `POST /hook`
- 低 QPS（hook 触发频率 = claude 工具调用频率，个位数/s 量级）
- LinkShell 生产验证用的就是原生 http [CITED: /tmp/linkshell-research/packages/cli/src/runtime/bridge-session.ts:873]
- 加 express 会多一个 dependency，增加 bundle size，没有任何收益

Node `http.Server` 的 keep-alive 默认 timeout：Node 20+ 是 5s（`server.keepAliveTimeout`），request timeout 默认 0（不超时）。**关键点**：PermissionRequest 要挂住 24h，单个 HTTP request 的 socket 必须能活 24h。
- 默认 `server.requestTimeout = 0` 意味着 Node 不会主动切单个 request 的连接 [CITED: nodejs.org/api/http.html]
- 但要显式 `server.setTimeout(0)` 或在 socket 级别 `req.setTimeout(0)`，否则某些 Node 子版本默认值可能变
- curl 这边用 `--max-time 86400` 控制客户端超时（CONTEXT.md 已写入模板）

### Pattern: settings.json atomic rename

**Why:** settings.json 没有官方并发保证 [CITED: code.claude.com/docs/en/hooks]，in-place 覆写如果 crash 会留下半写文件。

```typescript
// apps/proxy/src/serve/settings-injector.ts
import { readFileSync, writeFileSync, renameSync, existsSync } from "node:fs";
import { join } from "node:path";
import { homedir } from "node:os";

const SETTINGS_PATH = join(homedir(), ".claude", "settings.json");

export function injectClaudeHooks(port: number, daemonId: string): void {
  let existing: Record<string, unknown> = {};
  if (existsSync(SETTINGS_PATH)) {
    try {
      existing = JSON.parse(readFileSync(SETTINGS_PATH, "utf8"));
    } catch (err) {
      serviceLogger.warn({ err: String(err) }, "Failed to parse existing settings.json, start fresh");
      existing = {};
    }
  }

  const hooks = (existing.hooks ?? {}) as Record<string, unknown[]>;
  const curlCmd =
    `curl -s -X POST --max-time 86400 ` +
    `"http://127.0.0.1:${port}/hook?sessionId=$CC_ANYWHERE_SESSION_ID&event=$CLAUDE_HOOK_EVENT" ` +
    `--data-binary @- ; # cc-anywhere-hook-marker:${daemonId}`;

  const defaultEntry = { matcher: "", hooks: [{ type: "command", command: curlCmd, timeout: 5 }] };
  const permEntry = { matcher: "", hooks: [{ type: "command", command: curlCmd, timeout: 86400 }] };

  const eventMap: Record<string, typeof defaultEntry> = {
    PreToolUse: defaultEntry,
    PostToolUse: defaultEntry,
    PermissionRequest: permEntry,
    Stop: defaultEntry,
    UserPromptSubmit: defaultEntry,
    SessionStart: defaultEntry,
  };

  for (const [event, entry] of Object.entries(eventMap)) {
    let arr = Array.isArray(hooks[event]) ? hooks[event] : [];
    // 先剥离同 daemonId 的历史条目（重启场景幂等）
    arr = arr.filter((e) => !JSON.stringify(e).includes(`cc-anywhere-hook-marker:${daemonId}`));
    arr.push(entry);
    hooks[event] = arr;
  }

  const merged = { ...existing, hooks };
  const tmp = SETTINGS_PATH + ".tmp";
  writeFileSync(tmp, JSON.stringify(merged, null, 2));
  renameSync(tmp, SETTINGS_PATH); // POSIX atomic on same filesystem
}

export function cleanupClaudeHooks(daemonId: string): void {
  if (!existsSync(SETTINGS_PATH)) return;
  let parsed: Record<string, unknown>;
  try {
    parsed = JSON.parse(readFileSync(SETTINGS_PATH, "utf8"));
  } catch {
    return;
  }
  const hooks = parsed.hooks as Record<string, unknown[]> | undefined;
  if (!hooks) return;

  let changed = false;
  for (const [event, arr] of Object.entries(hooks)) {
    if (!Array.isArray(arr)) continue;
    const filtered = arr.filter((e) => !JSON.stringify(e).includes(`cc-anywhere-hook-marker:${daemonId}`));
    if (filtered.length !== arr.length) {
      changed = true;
      if (filtered.length === 0) delete hooks[event];
      else hooks[event] = filtered;
    }
  }
  if (Object.keys(hooks).length === 0) delete parsed.hooks;

  if (changed) {
    const tmp = SETTINGS_PATH + ".tmp";
    writeFileSync(tmp, JSON.stringify(parsed, null, 2));
    renameSync(tmp, SETTINGS_PATH);
  }
}
```

**关键不变量**：
- `writeFileSync(tmp) + renameSync` 是 POSIX atomic；claude file watcher 永远看到完整文件
- `JSON.stringify(entry).includes(marker)` 做 filter，依赖 marker 字符串在 command 行内唯一，不会误伤用户自定义 hook
- daemonId 不变跨 serve 重启（持久化在 `~/.cc-anywhere/daemon-id`）→ 同一 daemonId 的新旧条目都被 filter 掉再重新 append，幂等

### Pattern: Hook endpoint with挂住的 PermissionRequest

```typescript
// apps/proxy/src/serve/hook-endpoint.ts
import { createServer, type Server, type IncomingMessage, type ServerResponse } from "node:http";
import { nanoid } from "nanoid";
import { serviceLogger as log } from "../common/logger.js";
import type { HookApprovalManager } from "./hook-approval-manager.js";
import type { RelayConnection } from "./relay-connection.js";
import { buildMessage } from "@cc-anywhere/shared";

interface HookEndpointDeps {
  approvalManager: HookApprovalManager;
  relay: RelayConnection;
}

export async function startHookEndpoint(deps: HookEndpointDeps): Promise<{ port: number; close: () => Promise<void> }> {
  const server = createServer((req, res) => handleRequest(req, res, deps));
  server.setTimeout(0); // PermissionRequest 要挂 24h

  const port = await new Promise<number>((resolve, reject) => {
    server.once("error", reject);
    server.listen(0, "127.0.0.1", () => {
      const addr = server.address();
      if (typeof addr === "object" && addr) resolve(addr.port);
      else reject(new Error("Failed to get hook endpoint port"));
    });
  });

  log.info({ port }, "Hook endpoint listening");
  return {
    port,
    close: () =>
      new Promise((resolve) => {
        server.close(() => resolve());
      }),
  };
}

function handleRequest(req: IncomingMessage, res: ServerResponse, deps: HookEndpointDeps): void {
  if (req.method !== "POST" || !req.url?.startsWith("/hook")) {
    res.writeHead(404).end();
    return;
  }
  const url = new URL(req.url, "http://127.0.0.1");
  const sessionId = url.searchParams.get("sessionId");
  if (!sessionId) {
    res.writeHead(400).end("missing sessionId");
    return;
  }

  let body = "";
  req.on("data", (chunk: Buffer) => { body += chunk.toString(); });
  req.on("end", () => {
    let event: Record<string, unknown>;
    try {
      event = JSON.parse(body);
    } catch (err) {
      log.warn({ err: String(err), bodyLen: body.length }, "Hook body parse failed");
      res.writeHead(200).end("ok"); // 不阻塞 claude
      return;
    }

    const hookEventName = String(event.hook_event_name ?? "");
    const toolName = typeof event.tool_name === "string" ? event.tool_name : undefined;
    const toolInput = (event.tool_input && typeof event.tool_input === "object")
      ? event.tool_input as Record<string, unknown>
      : undefined;
    const transcriptPath = typeof event.transcript_path === "string" ? event.transcript_path : undefined;

    if (hookEventName === "PermissionRequest") {
      const requestId = nanoid(12);
      deps.approvalManager.register(requestId, sessionId, (decision) => {
        const responseBody = JSON.stringify({
          hookSpecificOutput: {
            hookEventName: "PermissionRequest",
            decision: { behavior: decision },
          },
        });
        res.writeHead(200, { "Content-Type": "application/json" }).end(responseBody);
      });

      // request 被断开（client 消失 / timeout）→ cleanup pending
      req.on("close", () => {
        deps.approvalManager.drop(requestId, "request closed");
      });

      // push envelope
      deps.relay.sendEnvelope(buildMessage(
        "terminal.permission_request",
        sessionId,
        Date.now(),
        { requestId, sessionId, toolName, toolInput, transcriptPath },
        "proxy",
      ));
      return;
    }

    // 其他事件：立刻 200 + 推 envelope
    res.writeHead(200).end("ok");

    deps.relay.sendEnvelope(buildMessage(
      "terminal.hook_event",
      sessionId,
      Date.now(),
      { hookEventName, sessionId, toolName, toolInput, transcriptPath },
      "proxy",
    ));

    // D-10: phase derivation
    const phase = derivePhase(hookEventName);
    if (phase) {
      deps.relay.sendEnvelope(buildMessage(
        "terminal.status",
        sessionId,
        Date.now(),
        { sessionId, phase },
        "proxy",
      ));
    }
  });
}

function derivePhase(evt: string): "idle" | "thinking" | "tool_use" | "waiting" | "error" | null {
  switch (evt) {
    case "PreToolUse": return "tool_use";
    case "PostToolUse": return "thinking";
    case "Stop": return "idle";
    case "UserPromptSubmit": return "thinking";
    case "SessionStart": return "idle";
    default: return null;
  }
}
```

### Pattern: hook-approval-manager (复刻 tool-approval-manager 结构)

```typescript
// apps/proxy/src/serve/hook-approval-manager.ts
import { serviceLogger as log } from "../common/logger.js";

type Decision = "allow" | "deny";
type Resolver = (decision: Decision) => void;

interface Pending {
  sessionId: string;
  resolve: Resolver;
  registeredAt: number;
  timer: NodeJS.Timeout;
}

const APPROVAL_TIMEOUT_MS = 24 * 60 * 60 * 1000;

// 与 ToolApprovalManager 对称：纯数据管理器，不做 HTTP。
// HTTP response 的 close 由调用方（hook-endpoint）管。
export class HookApprovalManager {
  private pending = new Map<string, Pending>();

  register(requestId: string, sessionId: string, resolve: Resolver): void {
    const timer = setTimeout(() => {
      const entry = this.pending.get(requestId);
      if (!entry) return;
      this.pending.delete(requestId);
      log.warn({ requestId, sessionId }, "Hook permission request timed out (24h), auto-deny");
      entry.resolve("deny");
    }, APPROVAL_TIMEOUT_MS);

    this.pending.set(requestId, { sessionId, resolve, registeredAt: Date.now(), timer });
  }

  resolve(requestId: string, decision: Decision): boolean {
    const entry = this.pending.get(requestId);
    if (!entry) return false;
    this.pending.delete(requestId);
    clearTimeout(entry.timer);
    entry.resolve(decision);
    return true;
  }

  drop(requestId: string, reason: string): void {
    const entry = this.pending.get(requestId);
    if (!entry) return;
    this.pending.delete(requestId);
    clearTimeout(entry.timer);
    log.info({ requestId, sessionId: entry.sessionId, reason }, "Hook permission dropped without decision");
    // 不 resolve：HTTP response 已被 req.on("close") 断开，写了也没人读
  }

  drainSession(sessionId: string): void {
    for (const [requestId, entry] of this.pending) {
      if (entry.sessionId !== sessionId) continue;
      this.pending.delete(requestId);
      clearTimeout(entry.timer);
      entry.resolve("deny");
      log.info({ requestId, sessionId }, "Session ended, pending hook permission auto-denied");
    }
  }

  drainAll(): void {
    for (const [requestId, entry] of this.pending) {
      clearTimeout(entry.timer);
      entry.resolve("deny");
      log.info({ requestId }, "Daemon shutting down, pending hook permission auto-denied");
    }
    this.pending.clear();
  }
}
```

### Anti-Patterns to Avoid

- **不要在 terminal.ts 起 HTTP server**。每 terminal 一份 endpoint 会导致多终端同时打开时 settings.json 被来回覆写，且每个 endpoint 的 port 都要写进 settings。D-01 已排除此路径。
- **不要用 `writeFileSync` 直接覆写 settings.json**。无 atomic 保证，crash 会留坏文件。必须 tmp + rename。
- **不要依赖 `process.exit` 前 `unlinkSync` 清理 marker**。SIGKILL 绕过一切 handler；必须在下次启动时按 daemonId filter 掉自己的旧条目作为兜底。
- **不要在 hook handler 里做重活**（日志大对象序列化、synchronous fs scan）。D-14: 非 PermissionRequest 必须 <50ms 返回，否则拖慢 claude。
- **不要复用 `tool-approval-manager`**。JSON 模式的 requestId 命名空间和 lifecycle 完全不同（worker socket vs HTTP response）；混在一起会让 cleanup 语义糊掉。D-12 已排除。
- **不要把 `transcriptPath` 当敏感信息打码**。它是 `~/.claude/projects/...` 下的 jsonl 路径，本来就是给 client 看的（未来 phase 可能要读）。

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP server | custom net.Server + HTTP parsing | `node:http.createServer` | built-in, 覆盖所有 HTTP/1.1 corner cases |
| 挂住 response 24h | 自己写 timer + poll | `http.ServerResponse` 直接不调用 `res.end()` + `req.on("close")` 监测断开 | Node http 原生支持长连接；只要不调 `end()` 就一直挂 |
| Atomic file write | `writeFile` + flush + fsync 手写 | `writeFileSync(tmp) + renameSync` | POSIX rename 是 atomic guarantee，跨文件系统才要 fsync |
| 唯一 ID 生成 | `Date.now() + Math.random()` 拼 | `nanoid(12)` | 项目已有；Math.random 碰撞在高频下概率非零 |
| JSON schema 校验 | 手写 typeof 检查 | `zod` discriminatedUnion | 项目已建立模式；编译期 + 运行期双重约束 |

**Key insight**: Node 标准库 + nanoid + zod 足够，不引任何新依赖。

## Runtime State Inventory

本 phase **不是 rename/refactor/migration**，是新增 feature。但确实引入了多处**外部可变运行时状态**，同样需要在 plan 中覆盖到：

| Category | Items | Action Required |
|----------|-------|------------------|
| Stored data | 无 | — |
| Live service config | `~/.claude/settings.json.hooks.{EventName}[]` —— claude 启动时读，file watcher 热更新 | **写**: serve 启动 atomic inject；**清**: serve 退出按 marker filter；**兜底**: serve 启动时按 daemonId filter 掉旧条目 |

| Build artifacts | 无 | — |

## Common Pitfalls

### Pitfall 1: settings.json 的 `hooks.PermissionRequest` 只在 Claude Code v? 之后存在

**What goes wrong:** 老版本 Claude Code CLI 可能不识别 `PermissionRequest` hook event，写进去 silently ignored，审批永远触发不到。

**Why it happens:** Claude Code hook API 仍在演进 [CITED: code.claude.com/docs/en/hooks]。官方文档列出的 events 是当前版本的全集，没有版本注解。

**How to avoid:**
- 文档里写清楚"需要 Claude Code v2+"（具体版本在第一次 E2E 时验证）
- D-15 已要求 `hookAvailable: boolean` 状态位告知 client；如果发现 PermissionRequest 从未 fire 但 PreToolUse 正常，降级为"hook 不可用于审批，走 OSC signal"
- 不把 `hookAvailable` 判断逻辑硬编码到本 phase；11.7 或更晚再加主动探测

**Warning signs:** 手机点 approve 没反应；claude 继续走 TUI 审批流程。

### Pitfall 2: hook 子进程 env 继承断链

**What goes wrong:** `pty.spawn(env: {...})` 注入的环境变量在 claude 里可读，但 hook 命令 `curl ... sessionId=$CC_ANYWHERE_SESSION_ID` 里拿不到。

**Why it happens:** claude 执行 hook command 用的是 `sh -c <command>`。shell 的变量展开依赖 env 是否 export 给子进程。Claude Code 文档明确 "Hooks inherit Claude Code's environment" [CITED: code.claude.com/docs/en/hooks]，说明 claude 会 pass full env 给 hook 子进程。POSIX `exec` 族默认 pass parent env，所以这条链路 **理论上成立**。

**How to avoid:**
- 第一个集成测试就是验证这条链。用 `--debug hooks` flag 启动 claude 看 hook stdout/stderr（见 Claude Code 文档）
- 备案：如果真不继承，把 port/sessionId 写进 hook command hardcode（牺牲每次 serve 重启后 settings.json 都要重写，但依然正确）；但这需要 per-terminal 重写 settings，成本高。
- 实测一下就知道，不要停留在假设层。

**Warning signs:** hook endpoint 收到 `sessionId=` 空字符串或 `$CC_ANYWHERE_SESSION_ID` 字面值。

### Pitfall 3: Node `http.Server` 的 keep-alive timeout 切断 24h 挂住的 request

**What goes wrong:** PermissionRequest 挂了 6 分钟，socket 被 Node 自动 close，claude 这边 curl 收到 connection reset，hook 无法响应 allow/deny。

**Why it happens:** Node `server.keepAliveTimeout` 默认 5 秒（空闲 keep-alive 连接的超时）。但**对于一个还没 end 的 request，请求级 timeout `server.requestTimeout` 才是相关值**，Node 20+ 默认是 0（不超时）。不过 socket-level `server.timeout` 默认 2 分钟（Node <13 是 2 分钟，后面的版本文档说是 0，但实测历史版本有反复）。

**How to avoid:** 显式设置，不依赖默认值：
```typescript
server.setTimeout(0);     // disable socket timeout
server.keepAliveTimeout = 0;  // belt-and-suspenders
server.requestTimeout = 0;    // for safety
```

**Warning signs:** 手机端点 Approve 后 claude 出现 `curl: connection reset by peer` 或 `timeout`。

### Pitfall 4: 多次 serve 重启污染 settings.json

**What goes wrong:** crash → restart → crash → restart，每次都 append，settings.json hooks 数组疯长。

**Why it happens:** 如果只在正常 SIGTERM 时 cleanup，crash 路径从不 cleanup。append 逻辑里如果没有"先 filter 自己 daemonId 的旧条目"步骤，就会堆积。

**How to avoid:** inject 函数每次必须 **先 filter 自己 daemonId 的旧条目再 append**（见 Pattern § settings.json atomic rename 的代码示例）。daemonId 跨重启不变 → 自己的旧条目能被识别 → 幂等。

**Warning signs:** `jq '.hooks.PermissionRequest | length' ~/.claude/settings.json` 随时间增长。

### Pitfall 5: proxy→client allowlist 忘了更新

**What goes wrong:** envelope 到 relay 静默被丢，client 一个事件都收不到，以为是 hook 没 fire。

**Why it happens:** relay 对 proxy→client control messages 有白名单 [VERIFIED: apps/relay/src/handlers/proxy.ts:10 PROXY_TO_CLIENT_TYPES]。新增类型必须加。Memory `project_relay_control_allowlist` 明确记录此教训。

**How to avoid:**
- `PROXY_TO_CLIENT_TYPES` 加 3 个（`terminal.hook_event`、`terminal.permission_request`、`terminal.status`）
- `proxy-to-client-types.test.ts` 的 `EXPECTED_TYPES` 跟着加，测试会守住这条
- `terminal.permission_decision` 是 **client → proxy**，不在此 allowlist

**Warning signs:** 单测通过但 E2E 无事件；relay 日志里 "dropped message: unknown type"。

### Pitfall 6: envelope type 命名冲突

**What goes wrong:** `terminal.status` 在 LinkShell 里已有含义（terminal 列表的状态）；本仓目前没有，但未来若加会冲突。

**Why it happens:** 命名太 generic。

**How to avoid:** 用 D-08 的原字面值 `terminal.status`。冲突概率低（本仓无 terminal.list / terminal.status 历史），而且跟 CONTEXT.md 锁死的事件名对齐。如果 planner/executor 发现本仓已有同名，改为 `terminal.hook_status`（加前缀，更明确），更新 CONTEXT.md。

## Hook Event Payload Schema (Reference)

[CITED: https://code.claude.com/docs/en/hooks]

所有 hook 共有字段：

```json
{
  "session_id": "abc123",
  "transcript_path": "/path/to/transcript.jsonl",
  "cwd": "/current/working/directory",
  "permission_mode": "default|plan|acceptEdits|auto|dontAsk|bypassPermissions",
  "hook_event_name": "PreToolUse|..."
}
```

**Note**: `session_id` 在 payload 里是 claude 自己的 session_id，不是 CC Anywhere 的。CC Anywhere 的 sessionId 走 URL query `?sessionId=...`（D-02）。

事件特有字段：

| Event | Key Fields (除共有字段外) |
|-------|---------------------------|
| `PreToolUse` | `tool_name`, `tool_input`, `tool_use_id` |
| `PostToolUse` | `tool_name`, `tool_input`, `tool_response`, `tool_use_id` |
| `PermissionRequest` | `tool_name`, `tool_input`, `permission_suggestions[]` |
| `Stop` | (none extra) |
| `UserPromptSubmit` | `prompt` |
| `SessionStart` | `source` (`startup|resume|clear|compact`), `model` |

**Timeout unit**: seconds [CITED: code.claude.com/docs/en/hooks]. 我们写 `timeout: 86400`（秒 = 24h）符合 D-05。

## Permission Response Protocol

**Hook stdout (exit code 0)** must be:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow"
    }
  }
}
```

或 `"behavior": "deny"` + 可选 `"message": "reason"` [CITED: code.claude.com/docs/en/hooks]。

**关键：exit code 2 才 block**（非 0 且非 2 是 non-blocking error）。我们的 curl 在 HTTP 200 时 exit 0，fold stdout 就是 claude 要的 JSON。**curl 命令不需要任何 jq 转换**，直接 `--data-binary @-` 发 payload，claude 收到的是 HTTP body（即我们 `res.end(responseBody)` 写的内容）作为 hook 的 stdout。

## Open Questions

1. **`hookAvailable` 状态位怎么暴露给 client**
   - 什么是知道的: D-15 要求 `bridge_status` 加布尔字段
   - 什么不清楚: bridge_status 当前是 IPC message（terminal ↔ serve），不是 envelope（serve ↔ relay ↔ client）。是扩展现有 bridge_status，还是新加一个 client 可见字段？
   - Recommendation: 在 `buildMessage("session_sync", ...)` 里 per-session 加 `hookAvailable`（与 inject 成功/失败 1:1 对应），或在 `terminal.status` 首个 envelope 附带。Planner 决策。

2. **多终端 attach 到同一 session 时 pendingHookPermission 的去重**
   - 什么是知道的: 本 phase 只要求临时简版 modal
   - 什么不清楚: 两个 browser tab 看同一 session，都会收到 permission_request；两边都能点 Approve/Deny 吗？第一个响应 take，第二个响应 drop？
   - Recommendation: take-first-win —— hook-approval-manager.resolve 返回 bool，已 taken 的 requestId 后续 decision 直接忽略。前端收到 `terminal.permission_decision` 广播（如果 relay 会广播回 client，11.7 之前可能不会）时清 modal；暂不做主动清理，让 requestId 过期自然结束。

3. **Claude Code 最低版本要求**
   - 什么是知道的: 官方文档的 PermissionRequest 是当前版本
   - 什么不清楚: 引入此特性的确切版本
   - Recommendation: 第一个集成测试在真实 claude CLI 上跑，记录当时版本，写入 README 和 CONTEXT.md 的 constraint 区。

## Security Domain

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | HTTP endpoint 只监听 127.0.0.1，无跨机访问 |
| V3 Session Management | no | — |
| V4 Access Control | partial | `sessionId` URL query 作为 terminal 隔离；非现有 terminal 的 sessionId 发来的 hook 应该 drop（验证清单） |
| V5 Input Validation | yes | hook payload JSON.parse 失败不 throw，log + ok ack；envelope 走 zod 校验 |
| V6 Cryptography | no | — |

### Known Threat Patterns

| Pattern | STRIDE | Mitigation |
|---------|--------|-----------|
| 本机另一用户进程向 127.0.0.1:port 发伪造 hook | Spoofing | macOS/Linux 下本机所有用户都能连 127.0.0.1，攻击面存在；信任本机进程是项目基线假设（proxy 本来就在用户机器跑）；可选加固：URL query 带 shared secret (daemonId 本身可当，因为只有 settings.json 和内存有) — 下个版本再考虑 |
| settings.json 被另一工具并发写导致条目丢失 | Tampering | atomic rename + marker filter；仍是**后写者赢**，但此场景下用户会看到 claude 不 fire 我们的 hook，重启 serve 即恢复 |
| hook body 过大 DoS | DoS | `req.on("data")` 里累 body 不做限制，考虑加 10MB cap；hook 事件对象正常很小 (<10KB) |
| PermissionRequest 挂死 claude | DoS | D-16 24h auto-deny 兜底 |
