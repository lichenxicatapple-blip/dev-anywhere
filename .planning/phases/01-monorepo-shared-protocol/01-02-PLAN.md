**builders/index.ts:**
```typescript
import { nanoid } from "nanoid";
import type { MessageEnvelope } from "../schemas/envelope.js";
import { MessageEnvelopeSchema } from "../schemas/envelope.js";

let sequenceCounter = 0;

export function createSequenceId(): number {
  return sequenceCounter++;
}

export function resetSequenceCounter(value = 0): void {
  sequenceCounter = value;
}

export function buildMessage<T extends MessageEnvelope["type"]>(
  type: T,
  sessionId: string,
  payload: Extract<MessageEnvelope, { type: T }>["payload"],
  source: "proxy" | "client",
): Extract<MessageEnvelope, { type: T }> {
  const envelope = {
    seq: createSequenceId(),
    sessionId,
    type,
    payload,
    timestamp: Date.now(),
    source,
    version: "1.0",
  };
  return MessageEnvelopeSchema.parse(envelope) as Extract<MessageEnvelope, { type: T }>;
}
```

Note: nanoid is imported but not used directly in builders -- it will be used for sessionId generation in later phases. Keep the import in package.json dependencies but do not import in builders/index.ts to avoid unused-import lint errors. If nanoid is only needed for session IDs (created by proxy/relay, not shared), consider whether it should stay in shared's deps or move to proxy's deps. For now, keep it in shared since D-10 says shared contains "sequence number generator" and IDs may be generated here.

Actually, remove the nanoid import from builders/index.ts. The builder uses a simple integer counter for sequence numbers per the research pattern. nanoid stays in shared's package.json as a dependency for downstream use (session ID generation helper could be added later).
