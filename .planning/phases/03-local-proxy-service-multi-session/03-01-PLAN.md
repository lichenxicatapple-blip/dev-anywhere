    **Step 2: Create `apps/proxy/src/line-buffer.ts`**

    Implement a `LineBuffer` class extending `node:stream.Transform`. Per D-07 and D-08, this handles the fact that Node.js `data` events split lines arbitrarily.

    The class:
    - Maintains a `private buffer: string = ""`
    - In `_transform(chunk, encoding, callback)`: appends `chunk.toString()` to buffer, splits on `"\n"`, keeps the last (potentially incomplete) segment in buffer, pushes all complete non-empty lines
    - In `_flush(callback)`: if buffer has remaining non-empty content, push it, then clear buffer and call callback
    - Each pushed value is a string (objectMode output is NOT needed; push strings)

    Per RESEARCH.md Pattern 2, this mirrors the cc-connect `bufio.Scanner` approach translated to Node.js streams.

    **Step 3: Create `apps/proxy/src/ipc-protocol.ts`**

    Define IPC message types for client-service communication over Unix domain socket (per D-29, D-30).

    Define a zod schema `IpcMessageSchema` as a discriminated union on `type` field with these message types:
    - `session_create_request`: `{ type, name?: string, mode: "pty" | "json" }` -- client requests new session
    - `session_create_response`: `{ type, sessionId: string, error?: string }` -- service responds with session ID
    - `session_list_request`: `{ type }` -- client requests session list
    - `session_list_response`: `{ type, sessions: Array<{ id, mode, state, createdAt, name? }> }` -- service responds
    - `session_terminate_request`: `{ type, sessionId: string }` -- client requests session termination
    - `session_terminate_response`: `{ type, sessionId: string, success: boolean }` -- service confirms
    - `pty_register`: `{ type, sessionId: string }` -- client registers a PTY session with service
    - `pty_deregister`: `{ type, sessionId: string }` -- client deregisters PTY session (terminal close, per D-14)
    - `pty_output`: `{ type, sessionId: string, data: string }` -- PTY output forwarded from client to service (for future relay, per D-12)
    - `pty_input`: `{ type, sessionId: string, data: string }` -- input from service to client's PTY stdin (for future Feishu input, per D-13)
    - `heartbeat`: `{ type, sessionId?: string }` -- client heartbeat (per D-26)
    - `heartbeat_ack`: `{ type }` -- service heartbeat acknowledgment
    - `session_status_update`: `{ type, sessionId: string, state: SessionState }` -- service broadcasts state change (per D-16)
    - `error`: `{ type, message: string, code?: string }` -- error response

    Export `type IpcMessage = z.infer<typeof IpcMessageSchema>`.

    Export `serializeIpc(msg: IpcMessage): string` that returns `JSON.stringify(msg) + "\n"` (NDJSON framing per RESEARCH Pattern 2).

    Export `createIpcReader(stream: NodeJS.ReadableStream, onMessage: (msg: IpcMessage) => void): void` that pipes the stream through a `LineBuffer`, then for each line: tries `JSON.parse`, validates against `IpcMessageSchema`, and calls `onMessage` with the validated result. Invalid lines are logged as warnings (not thrown) per D-07 pattern.

    Import `SessionState` from `@cc-anywhere/shared` for the state field in session messages.

  <behavior>
    PtyManager refactored exit behavior:
    - When child exits with exitCode 42, onSessionExit callback is called with 42
    - When child exits with signal 2, onSessionExit callback is called with 130 (128+2)
    - process.exit() is never called from PtyManager (neither in onExit nor in cleanup)
    - cleanup() kills child and restores raw mode but does NOT call process.exit()
    - cleanup() calls onSessionExit callback with the given code
    - If no onSessionExit callback is set, child exit is still handled gracefully (no crash)
    - Signal handlers (SIGTERM, SIGHUP) call cleanup which invokes onSessionExit
    - PtyManager no longer registers global process event handlers (uncaughtException, unhandledRejection, SIGTERM, SIGHUP) -- these are the caller's responsibility in multi-session mode
  </behavior>
  <action>

    3. In `cleanup(exitCode)` method (currently line 109-125): Remove `process.exit(exitCode)` at the end. Instead, call `this.onSessionExit?.(exitCode)` after killing child and restoring raw mode.

In multi-session mode, global signal/exception handling is the caller's responsibility. PtyManager should only manage its own PTY lifecycle.
