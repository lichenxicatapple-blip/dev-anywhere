<objective>
Close three audit-confirmed gaps from Phase 10 post-verification:

1. **Message subscription coverage gap** (root cause of "暂无会话" bug): Proxy pushes `session_list` / `session_status` / `pty_state` envelopes, relay forwards, but web client has no subscriber. Create a global `session-dispatcher.ts` mirroring the `chat-dispatcher` pattern and register it in `use-relay-setup.ts`. The session-store already has the target actions (`setSessions`, `updateSessionState`, `setHistorySessions`); only the wire-up is missing.

2. **History feature UI not migrated** from Feishu: Types / store slice / single-session message replay are all in place, but list-level handling (`session_history_response` subscription), UI (group-by-projectDir + Resume button), and the `session_history_request` trigger are absent. Migrate UI and dispatch logic from `apps/feishu/src/pages/session-list/index.tsx`.

3. **Phase-machine tests lost** in migration: Feishu has 387 lines / ~27 test cases covering ws status changes × phase matrix + relay message handlers + reconnect timer semantics + cold start recovery. Web has zero. Port them with minimal structural changes (feishu uses reducer+dispatch pattern, web uses zustand getState/setState; wrap store access behind the same createTestEnv helper).

4. **Exception E2E absent**: No end-to-end test for proxy offline→online transitions, session terminate while in chat, ws reconnect while chatting. Add `session-lifecycle.spec.ts` covering these three scenarios at minimum.

**Also in scope (5th audit finding):** Cross-check remaining feishu dispatch subscriptions (permission_mode_change_response, session_terminate_response, dir_list_response, session_create_response) against web coverage. Migrate anything currently missing that blocks a UI flow.

Why now: "暂无会话" bug today proved UI review without subscription audit is a false green. Every subsequent verification needs the underlying data pipeline trustworthy first.

Output: 1 new global dispatcher, 1 new history UI component, 1 new unit test file (≥27 cases matching feishu coverage), 1 new e2e spec (≥3 scenarios), updated wiring in session-list / create-session-dialog / use-relay-setup.
</objective>

<threat_model>
- **Risk A**: `session-dispatcher` and `chat-dispatcher` may both subscribe to same ws instance and race on shared message types. Mitigation: dispatchers partition by `msg.type` exhaustively; each type has at most one handler. Document ownership in head-of-file comment.
- **Risk B**: `session_list` envelope arrives before `proxy_select_response` (race); setSessions on unknown proxy. Mitigation: session-dispatcher short-circuits when `selectedProxyId` is null.
- **Risk C**: Phase-machine tests port wholesale but zustand-vs-reducer idiom forces rewrite. Mitigation: extract `createTestEnv(phase)` helper returning `{ getState, setStateForTest, ... }` that wraps `useAppStore.setState()` so test body reads like feishu's.
- **Risk D**: Resume from history creates a new cc-anywhere session bound to `resumeSessionId`, but Claude Code jsonl might have been rotated / deleted. Error path: session_create_response error. Mitigation: show error toast, preserve history list, do not clear dialog.
- **Risk E**: Duplicate subscription register if `useRelaySetup` effect re-runs under StrictMode / HMR. Mitigation: `registerSessionDispatcher` returns an unsub; cleanup calls it.
- **Risk F**: E2E specs flake because they depend on real relay+proxy. Mitigation: mock ws messages via `page.evaluate` pushing to the relay client directly (pattern used in follow-output.spec.ts), not real backend.
</threat_model>
